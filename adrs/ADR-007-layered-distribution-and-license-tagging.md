# ADR-007: Layered distribution and per-extension license tagging

Status: Accepted
Date: 2026-05-06
Deciders: John Edge (CTO)
Supersedes: none

Note: This ADR was originally written using "Pi" as the platform name. Per ADR-009, the platform is renamed to Mirepoix; this ADR's architectural commitments are unchanged.

## Context

Mirepoix started as a single project — Kavara's coding agent harness — but the production shape we have committed to building is materially different. Mirepoix is a *distribution* of a Software-3.0 operating system, with Kavara as the maintainer of one variant and customer-facing variants riding on top of it. The Mirepoix metaphor that we adopted as the architectural framing — base plus Kavara-Mirepoix plus Customer-X-Mirepoix — is the same shape every successful operating-system distribution model has taken. The Linux kernel plus Ubuntu plus Ubuntu-for-IoT pattern. The BSD kernel plus macOS plus macOS-Server pattern. The challenge each of these distributions faced is the same one we now face: at the *extension* level, how do you mix code that should be public, code that is licensed to specific customers, and code that should never leave the maintainer's organization, without leaking the wrong category to the wrong audience?

Our prior decisions in ADR-003 and ADR-006 settled on NPM as the distribution channel for Mirepoix extensions and rejected building a Kavara-controlled marketplace. That posture is correct for the Mirepoix-base layer, where the extensions are open-source by intent. It does not, by itself, settle the question for the Kavara-Mirepoix layer or the Customer-X-Mirepoix layer, where extensions encode Kavara's engineering taste, customer-specific runbooks, and IP that the Model-as-a-Service business depends on.

The CTO direction is unambiguous: conservative by default, opt-in to public on a case-by-case basis. Every extension is treated as Kavara-confidential unless its author has explicitly tagged it for release. Promotion to public is a deliberate act, never a default and never silent. This ADR specifies the architectural mechanism that enforces that posture and the layering that makes it operate at the bundle level rather than at the per-package level.

## Decision

Mirepoix adopts a three-layer distribution model and a per-extension license-tagging contract enforced by the bundler at build time.

The first layer is **Mirepoix-base**. The four packages from ADR-001 (`@mirepoix/ai`, `@mirepoix/core`, `@mirepoix/coding`, `@mirepoix/cli`) plus the small set of reference extensions in the repository's `examples/` directory. Mirepoix-base is fully public, MIT-licensed, and published to public NPM. It carries no Kavara identity beyond authorship attribution. Anyone — including non-Kavara users — can install Mirepoix-base and operate it as a generic coding agent harness. This is the kernel-and-drivers layer.

The second layer is **Kavara-Mirepoix**. A curated bundle of extensions, skills, system-prompt fragments, footer configurations, themes, and routing policies that turn the open Mirepoix-base into the harness Kavara engineers actually run. This is the Kavara distribution of the Mirepoix OS. It includes the KServe-on-OCP provider, the multi-persona architecture-review extension, the AGENTS.md template tuned for Kavara conventions, the connectors that know about Kafka topics like `market.equities.trades` and `ulysses.tensor.frames.*`, the substrate-aware skills for GCP Kata-VM and Azure TDX postures, and the agent-team configurations that mirror Kavara's actual roles. Kavara-Mirepoix is versioned independently and tagged with semver (e.g., `kavara-mirepoix@0.1.0`).

The third layer is **Customer-X-Mirepoix**. A per-customer remix of Kavara-Mirepoix, layered with customer-specific extensions, skills, and configs. These instances ship alongside the Kavara MaaS appliance to federal POC sites, Databank DL360 deployments, and similar customer-facing engagements. They encode Kavara's engineering rigor in a form the customer (or Kavara's customer-success team operating on the customer's behalf) uses to operate the deployment. Each Customer-X-Mirepoix is a private repository referencing a specific Kavara-Mirepoix version plus its own customer-specific extensions.

Every extension declares license metadata in its `package.json` (or in an equivalent `mirepoix-extension.json` for path-loaded extensions) under the field `mirepoix.distribution`. Three distribution tags are supported.

`internal` is Kavara-confidential. Never publishable to public NPM. Loadable only from internal repositories or path-based references. The bundler refuses to include `internal`-tagged extensions in any bundle destined for a public registry or for unsigned customer distribution.

`customer-licensed` is distributable as part of a Kavara MaaS deliverable to a specific customer or class of customers, under Kavara's commercial license terms. Bundled into customer-deliverable artifacts (signed appliance bundles, per ADR-005's session-log integrity model) but not into public NPM publications.

`public` is explicitly approved for public release. Published to public NPM under `@kavara/mirepoix-extension-*` and visible in the public Mirepoix ecosystem. Promotion to `public` is an explicit, reviewable action — a manifest change committed under a CTO-level approval gate, not a default.

The default tag for every new extension is `internal`. An untagged extension is treated as `internal`. The bundler refuses to operate on an extension that is missing a tag and refuses to publish a bundle that contains untagged extensions. There is no quiet failure mode in which private code reaches a public registry because someone forgot a field.

The bundler is a single tool, `mirepoix bundle`, shipped as part of `@mirepoix/cli`. It takes a target — `public-npm`, `customer-deliverable`, or `internal` — and a manifest listing the extensions to include. It enforces the license-tag contract at build time. For target `public-npm`, every included extension must have tag `public`; the build fails loud and blocks if any extension has tag `internal` or `customer-licensed`. For target `customer-deliverable`, every included extension must have tag `public` or `customer-licensed`; the build fails if any extension has tag `internal`. Customer-deliverable bundles are additionally signed (per ADR-005's appliance-bundle integrity model) and the signature includes the manifest of license tags so that the customer can audit what they received. For target `internal`, any tag is allowed; internal bundles are not distributed outside Kavara, and the bundler does not push them to any public registry.

There is no implicit conversion between tags. An extension cannot be "temporarily public" or "soft-internal." A tag change requires a manifest change, a commit, and a review.

Mirepoix-base layer extensions all carry tag `public`. Kavara-Mirepoix layer extensions are tagged per-extension based on whether they encode Kavara IP that should be advertised externally — the architecture-review meta-agent, generic OpenShift connectors, the cascade-routing extension — versus IP that should not — Kirk-specific runbooks, customer-deployment skills, the substrate-matrix-aware substrate selectors. Customer-X-Mirepoix layer extensions default to `customer-licensed` for the customer they ship with, and `internal` for anything that should not even leave Kavara as part of a customer deliverable.

The legal license (MIT, Apache 2.0, Kavara Commercial) and the distribution tag are independent fields. An extension can be MIT-licensed legally and still tagged `internal` — we choose not to publish it. An extension can be Kavara-Commercial-licensed and tagged `customer-licensed` — we publish it under those terms to specific customers. Conflating the two creates confusion and we do not.

## Consequences

The first consequence is that the public surface of Kavara-Mirepoix grows deliberately. Every extension that becomes public is one that Kavara has explicitly chosen to release. This has marketing value — each public extension is advertising for Kavara's engineering taste, in line with the scarcity-shift thesis that taste is the rare resource — and it has IP value, because private extensions encode the Kavara-specific knowledge the MaaS business depends on. We accept that this is a slower release cadence than a permissive-default world; it is the right trade-off for a CTO-driven business with real IP at stake.

The second consequence is that we never have to ask whether a thing is accidentally public, because the bundler enforces tags at build time. A developer who writes a new Kavara-Mirepoix extension and does not tag it gets `internal` by default; the bundler refuses to publish a `public-npm` bundle that contains it. This is the same posture as `npm publish --access restricted` for scoped packages, but enforced at the bundle level rather than at the per-package level, and applied uniformly across all three distribution layers.

The third consequence is that `customer-licensed` becomes a meaningful distinct posture in our distribution model. A federal customer can receive a Customer-X-Mirepoix instance that contains Kavara-IP-bearing extensions under license, without those extensions ever entering a public registry. The signed bundle and the license-tag manifest give the customer an audit trail of what they got and on what terms. This composes cleanly with the appliance-bundle attestation flow that the SOR architecture review identified as Phase 4 work.

The fourth consequence is that promotion of extensions from `internal` to `public` becomes an explicit governance act. We will document the promotion workflow in the repository — what review is required, who can approve, how the change is recorded — but the architectural commitment in this ADR is that the act is explicit and visible. Promotion is not an accident.

The fifth consequence is that internal-only extensions need a non-public distribution channel. We use private GitHub repositories under the `UlyssesModel` and `kavara` orgs, with extensions installable via `git+ssh://github.com/UlyssesModel/...` URLs in `package.json`. We do not stand up a private NPM registry; the operational cost is not justified at our scale, consistent with ADR-006. For Kavara employees, repository access is controlled by GitHub team membership; for customer-licensed bundles, the bundle is the unit of distribution and the customer never sees the source of `internal` or `customer-licensed` extensions, only their compiled form.

The sixth consequence is that the Kavara-Mirepoix bundle becomes a versioned artifact in its own right. We tag releases with semver, and a customer-facing remix references a specific Kavara-Mirepoix version in its manifest. This gives us reproducibility at the bundle level — a Customer-X-Mirepoix installation a year from now can be rebuilt from its manifest and produce the same artifact, or upgraded by changing the Kavara-Mirepoix version reference and rebundling.

The seventh consequence is that we will need to be disciplined about what goes into each layer. A common failure mode for layered distributions is that extensions creep down — Kavara-specific code lands in Mirepoix-base because someone is in a hurry, customer-specific code lands in Kavara-Mirepoix for the same reason. We treat this as a code-review discipline issue, not an architectural one, and we do not build defensive structures into the codebase against it. The discipline is straightforward: extensions in Mirepoix-base must contain no Kavara-identifying content, and extensions in Kavara-Mirepoix must contain no customer-identifying content. Reviewers enforce this.

The eighth consequence is that the agent's ability to write extensions, per ADR-003, needs to be license-tag-aware. When the agent writes a new extension during a session, it tags the extension `internal` by default and surfaces this in the operator-visible output. If the operator wants the extension promoted to `public`, that is a separate, explicit action the operator takes after the extension is reviewed. The agent does not self-promote its own work to public.

The ninth consequence, drawing on the Software 3.0 framing, is that the autonomy slider in Mirepoix is realized not as a UI element but as a bundle-composition decision. Customer-X-Mirepoix instances ship with the autonomy preset appropriate to the customer's posture — a federal POC site with TDX appliance gets the `careful` preset (single agent, gate-hooks for damage control, til-done discipline), while a Kavara-internal engineering bundle gets the `trusted` preset (multi-agent teams, meta-extension-writer enabled). The license-tagging model and the bundle-composition model compose: the bundler picks both the autonomy level and the license posture from the manifest, in a single pass.

The tenth consequence is that we are committed to keeping `mirepoix bundle` simple. The bundler should be readable in one sitting — a few hundred lines of TypeScript — because it is the load-bearing tool that prevents IP leakage. We will not let it grow into a complex build system; if we need build-system features, we add them as separate commands rather than overload the bundler.

## Alternatives considered

We considered a permissive default — every extension is public unless explicitly tagged otherwise. This minimizes friction for the open-source workflow and treats Kavara-Mirepoix as a marketing surface. We rejected it because the asymmetry of the failure modes is unacceptable: a missed tag in a permissive-default world publishes private code, while a missed tag in a restrictive-default world keeps something private that should have been public. The first failure cannot be undone; the second is recoverable by adding the tag and re-publishing. Conservative-by-default is the only honest posture for a CTO-driven business with real IP.

We considered a per-customer remix model with no shared middle layer — every customer gets a fully bespoke Mirepoix build with no Kavara-Mirepoix in between. We rejected it because the duplication tax is enormous. Kavara-Mirepoix as a shared middle layer captures the common Kavara-flavored extensions once; customer remixes layer in just the customer-specific parts. This is the same argument that justifies any base-image-plus-overlay deployment model.

We considered hosting a private NPM registry (Verdaccio, JFrog, Sonatype) for internal extensions. We rejected it for the operational cost, in line with ADR-006's stance against running ecosystem infrastructure. GitHub-hosted private repositories with `git+ssh://` install URLs are sufficient for our scale.

We considered using a license-tagging vocabulary borrowed from existing ecosystems — SPDX identifiers, OSI-approved license names, or similar. We rejected it because SPDX vocabulary describes the *legal* license, while what we need to express is the *distribution posture*. An extension can be MIT-licensed (legally permissive) and still tagged `internal` (we choose not to publish it). The two are independent and conflating them creates confusion. Our three tags are about distribution; the legal license is a separate field.

We considered making every promotion to `public` require an automated CTO-level signature in code. We rejected the architectural commitment to that workflow because it is more naturally a governance question than an architectural one. The architecture supports any approval workflow; the governance layer decides who approves what. We will document the current governance policy in the repository, and we expect it to evolve as Kavara grows.

We considered automated detection of "looks Kavara-specific" content in extensions tagged `public` (string-search for customer names, internal hostnames, Kafka topic names, AMX-specific identifiers). We rejected it for the public-bundle build path because false negatives are catastrophic and false positives are cheap to handle by humans. We may add a non-blocking warning step that flags suspicious content in `public` extensions for human review, but it does not gate the build.

We considered four tags rather than three, with a fourth `partner-licensed` tier for shared development with companies like Red Hat that sit between "internal" and "customer-licensed." We rejected the fourth tag for now because the Red Hat collaboration is currently structured as outside-collaborator commits to the `tiberius-openshift` repository under Apache 2.0 (effectively `public`), and we do not have a current case requiring a partner-distinct tier. We may revisit if a future partner relationship needs it.

## Implementation notes

The license tag lives at the top level of `package.json` for NPM-distributed extensions and in an adjacent `mirepoix-extension.json` for path-loaded extensions, under the field `mirepoix.distribution` with values `internal`, `customer-licensed`, or `public`. The bundler reads this field directly. Extensions without the field are treated as `internal` and the bundler refuses to publish-target them.

The bundler is implemented as `mirepoix bundle <target> --manifest <manifest.yaml>` in `@mirepoix/cli`. The manifest is a YAML file listing the Kavara-Mirepoix version, the customer overlay (if any), the autonomy preset (referencing the slider-as-bundle-composition consequence above), and the extensions to include. The bundler resolves all extensions, validates their license tags against the target, signs the bundle for `customer-deliverable` targets, and publishes or writes the artifact.

The Kavara-Mirepoix public bundle monorepo lives at `github.com/UlyssesModel/kavara-mirepoix` (public). It hosts only `public`-tagged extensions, the public bundle manifest, the public-facing autonomy preset definitions, and the documentation. The fact that the repo is itself public is the case-by-case-opt-in policy made physical: code lands here only when an extension has been deliberately promoted to `public`. The repo's CI runs the bundler in `public-npm` mode and refuses to publish if any `internal` or `customer-licensed` extension has accidentally been committed.

`internal`-tagged extensions live in a parallel private repository under the same `UlyssesModel` GitHub organization (URL to be settled by a follow-up governance decision; the default working name is `kavara-mirepoix-internal`). Kavara engineers running Mirepoix locally compose their working bundle from both repositories — the public one for shared, externally-visible Kavara taste, the private one for Kavara-confidential extensions. The bundler resolves both repository sources via path or `git+ssh://` references in the manifest. There is no third "intermediate" repository; the line between "public" and "internal" is the line between the two repositories, and that physical separation is the additional defense against accidental leakage on top of the build-time tag enforcement.

Each extension in either repository is a workspace package with its own `package.json` and `mirepoix.distribution` field. New extensions in the public repo must be tagged `public` (the CI enforces this). New extensions in the private repo default to `internal` and may be tagged `customer-licensed` for inclusion in customer deliverables.

Customer-X-Mirepoix instances live in customer-specific private repositories — for example, `github.com/UlyssesModel/customer-leidos-mirepoix`, with access controlled by customer-specific GitHub teams. Each references a specific Kavara-Mirepoix version in its manifest and adds customer-specific extensions, all tagged `customer-licensed` or `internal`. The customer-deliverable bundle is built from this repository and signed before delivery.

The agent-written-extension path (per ADR-003) writes a `mirepoix-extension.json` adjacent to the new TypeScript module with `distribution: internal` set explicitly. The agent's worked-example documentation in `packages/mirepoix-coding/src/skills/writing-extensions.md` includes this requirement and shows the manifest format. Promotion to `public` is a follow-up operator action documented in the same skill file.

The license tag is enforced at build time, never at runtime. An `internal` extension loads and runs identically to a `public` one when Mirepoix is operating; the difference is only in what gets bundled and where the bundle goes. This keeps the runtime simple and the enforcement crisp.

The autonomy presets referenced in this ADR (`careful`, `trusted`, `autonomous`) are concrete named bundles defined in `kavara-mirepoix/presets/`. Each preset is a YAML file listing the extensions, skills, and gate-hooks that comprise it. The presets compose with the license-tagging model: a `careful` preset bundle for a customer-deliverable target will include only `public` and `customer-licensed` extensions, even if the preset's source manifest references additional `internal` ones. This composition is the bundler's responsibility, not the preset author's.
