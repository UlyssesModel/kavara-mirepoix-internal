# ADR-013: Codex as a teammate — orchestrated face-off in on-loop

Status: Accepted
Date: 2026-05-15
Deciders: John Edge (CTO)
Supersedes: none

## Context

Until now every step of Mirepoix's `on-loop` pipeline has been single-agent. When the orchestrator dispatches the architect, coding, testing, security, doc+build, or reviewer phase, exactly one Claude-based subagent runs and its verdict is the verdict. The pipeline is deterministic, auditable, and cheap, but it is also single-source-of-truth on a per-phase basis — the REVIEW gate in particular reflects one model's perspective on the diff, which is exactly the point in the pipeline where a second perspective has the most leverage.

OpenAI's `openai/codex-plugin-cc` plugin shipped (codex-cli 0.130.0 installed in the operator's Claude Code session) and adds a second teammate to the same session. The plugin exposes operator-invokable slash commands (`/codex:review`, `/codex:adversarial-review`, `/codex:rescue`, `/codex:setup`, `/codex:status`, `/codex:result`, `/codex:cancel`), a forwarding subagent (`codex:codex-rescue`) that calls the Codex CLI's `task` helper, and three internal skills (`codex-cli-runtime`, `codex-result-handling`, `gpt-5-4-prompting`) that the plugin's own commands consume. Critically, the `codex-result-handling` skill explicitly forbids auto-applying Codex review findings — Codex verdicts must be presented verbatim and the operator decides which (if any) to address. This forecloses one design choice (auto-apply) and shapes the rest of the verdict-composition surface. Notably, `codex-plugin-cc` is a Claude Code plugin installed in the operator's session, not a TypeScript extension loaded into the Mirepoix harness process — it sits outside the extension model of ADR-003 by design; the harness packages (`@mirepoix/ai`, `@mirepoix/coding`, `@mirepoix/core`, `@mirepoix/cli`) do not change.

This ADR commits to the design before sub-phase E (self-modification mechanics on the locked host) so the locked host inherits a designed posture rather than a retrofit. The locked host (scotty-gpu under ADR-010's deny-all-egress firewall) raises a real question about whether Codex's API path is reachable at all from inside the lockdown; that question is flagged as a sub-phase E concern in Consequences below and is not resolved here.

## Decision

ADR-013 makes five architectural commitments.

The first commitment is that the on-loop REVIEW phase dispatches Claude and Codex in parallel. When `/on-loop` reaches REVIEW, the orchestrator launches the existing `on-loop:review` Claude subagent and `/codex:adversarial-review` on the same branch at the same time, awaits both verdicts, and posts each verbatim into the PR description in dedicated sections. Either verdict can REQUEST_CHANGES and block merge; the gate is the union of both reviewers' concerns. Sequential dispatch (Claude first, then Codex only on Claude-approve) was considered and rejected because it discards Codex's perspective exactly when it would most catch a confirmation-biased Claude approval. The two reviewers use different source vocabularies (Claude `APPROVE` / `REQUEST_CHANGES`; Codex `approve` / `needs-attention` per the plugin's `review-output.schema.json`); the orchestrator normalizes both to a binary block/approve decision for gate purposes while preserving the body text of each verdict verbatim. The runbook (§4) carries the normalization table.

The second commitment is that CODE retry-exhaust dispatches `codex:codex-rescue` as a final attempt rather than dead-ending. After three Claude coding attempts have failed against TEST or SECURITY, the orchestrator dispatches the `codex:codex-rescue` subagent with the accumulated TEST and SECURITY feedback as input. The rescue output is NOT bypassed past TEST and SECURITY — it re-enters those phases like any other coding-agent diff. Additionally, the orchestrator captures the touched-file list from the rescue output (per the `codex-result-handling` skill), verifies the touched files are within the spec's diff allowlist, and dispatches a fresh REVIEW face-off on the rescue output. The rescue path does NOT skip any gate; if TEST, SECURITY, or REVIEW fails on the rescue output, the rescue diff is reverted and the PR cannot land without operator intervention. The escape valve is "another shot at coding under a second model with full gate containment", not "skip the gates".

The third commitment is that the architect's spec output adopts XML-block sections per the `gpt-5-4-prompting` skill that ships with `codex-plugin-cc`. Load-bearing structure moves into `<task>`, `<structured_output_contract>`, `<default_follow_through_policy>`, `<verification_loop>`, `<grounding_rules>`, and `<action_safety>` tags; markdown headers remain only inside tags for narrative readability. This tightens the contract with downstream agents whether or not Codex is in the dispatch chain, and is mandatory for any architect output whose downstream consumer is Codex (e.g., CODE retry-exhaust dispatch). The on-loop plugin's `architect.md` template is not patched in this PR (cross-repo; out of scope); the convention propagates via CLAUDE.md being auto-loaded as project context when the architect agent is dispatched.

The fourth commitment is a venue-split default. The Codex teammate posture differs between Mirepoix's two deployment venues per ADR-012:

- **Mirepoix-build venue (default).** Both teammates run per commitments (1)–(3). REVIEW dispatches Claude + Codex in parallel; CODE retry-exhaust dispatches `codex:codex-rescue`. This is the steady state for non-confidential `/on-loop` runs on `kavara-builder` and operator-workstation sessions.
- **Mirepoix-secure venue.** Codex is **disabled by default**. The locked host's deny-all-egress posture (ADR-010) precludes Codex's auth path to OpenAI from inside the lockdown. Enabling Codex on Mirepoix-secure requires a **superseding ADR that explicitly weakens ADR-010** — neither this ADR nor a runbook tweak is sufficient. Mirepoix-secure runs Claude-only review/code in v1 (via local Ollama serving Qwen2.5-Coder where the model provider is harness-local). The sub-phase E carve-out described in Consequences is the venue-specific instantiation of this rule, not an inheritance of the Mirepoix-build default.

The fifth commitment is that Codex-unavailable degrades gracefully to Claude-only on Mirepoix-build. If `/codex:status` reports Codex is not installed, not authenticated, or version-incompatible (codex-cli below 0.130.0), the orchestrator falls back to Claude-only for the affected phase and appends `[codex-unavailable] <reason>` to the session's `changes.log`. The PR is not blocked on tooling state, and the orchestrator does not auto-install Codex. Codex's presence is an operator-side prerequisite documented in the runbook, not a code-enforced harness dependency. On Mirepoix-secure this is the steady state per commitment (4): every REVIEW emits the `[codex-unavailable] mirepoix-secure-default` marker without probing `/codex:status`.

## Consequences

The first consequence is that the REVIEW gate is now parallel-two-agent by default. Wall-clock latency for REVIEW is the max of the two agents' runtimes rather than Claude's alone — typically a moderate increase, since `/codex:adversarial-review` is bounded by the same diff size. The verdict quality strictly improves: confirmation-biased single-model approvals are caught more often, and adversarial framing surfaces design failure modes that a friendly reviewer would soft-pedal. The cost of a parallel verdict is small; the cost of merging a confirmation-biased approval is not.

The second consequence is that CODE retry-exhaust is no longer a dead end. Before this ADR, three Claude failures against TEST or SECURITY ended the pipeline. After this ADR, the fourth attempt is `codex:codex-rescue` with the accumulated feedback as context. The gates are not weakened — TEST and SECURITY re-run against the rescue output — but the pipeline now has a designed second-model attempt before surrendering to operator intervention. Sub-phases that previously timed out may now land cleanly under a second model.

The third consequence is that architect output style changes. Prior architect notes used plain markdown H2/H3 headers throughout; new architect notes use XML-block tags for the load-bearing structure (see Decision #3). This is a one-time style shift, but it is mandatory rather than aspirational — coding, testing, security, and review agents that read architect notes will need to handle both styles transitionally. CLAUDE.md is the convention vehicle; a future cross-repo PR will patch the on-loop plugin's `architect.md` template directly so non-Mirepoix consumers benefit too.

The fourth consequence is a cross-provider operator dependency. The operator's Claude Code session must have `codex-plugin-cc` installed (`/plugin install codex@openai-codex`), Codex CLI on PATH (`command -v codex` exits 0), and an authenticated `/codex:setup`. The runbook (`docs/CODEX-TEAMMATE-RUNBOOK.md`) covers install and recovery. This dependency is operator-side, not a `package.json` addition — `@mirepoix/ai`, `@mirepoix/coding`, `@mirepoix/core`, and `@mirepoix/cli` do not gain a Codex provider slot or tool; that is sub-phase E or later territory.

The fifth consequence is the sub-phase E carve-out. Sub-phase E (Mirepoix-the-harness running its own on-loop on scotty-gpu) does **not** inherit the Mirepoix-build default. Per commitment (4), Codex is disabled on Mirepoix-secure by default — the locked host's deny-all-egress posture under ADR-010 precludes Codex's API path to OpenAI from inside the lockdown, and re-opening that path would require a superseding ADR that explicitly weakens ADR-010. Sub-phase E will land with Codex-disabled as the steady state for the locked host; any reconsideration (firewall allowlist carve-out, bastion-mediated tunnel, or other architectural change) is out of scope for this ADR and must be raised as a separate superseding ADR.

## Alternatives considered

- **Codex as default coder.** Rejected. Claude performs well on the happy path; reserving Codex for high-leverage cases (REVIEW adversarial framing, CODE retry-exhaust) preserves Codex's second-model value where it matters most. Making Codex the default coder would invert the cost/benefit and would create a continuous operator dependency on `codex-plugin-cc` being available for every coding attempt rather than just the rescue path.
- **Auto-apply Codex review findings.** Rejected. The `codex-result-handling` skill that ships with `codex-plugin-cc` explicitly forbids this — Codex verdicts must be presented verbatim and the operator decides which (if any) to address. CLAUDE.md encodes this as a hard "don't"; the runbook surfaces the rule under "What NOT to do".
- **Single-agent review (Codex-only).** Rejected. Replacing Claude with Codex as the sole REVIEW agent loses Claude's perspective and reintroduces single-point-of-model-bias. The face-off design specifically wants both perspectives.
- **Sequential REVIEW (Claude first, then Codex only if Claude approves).** Rejected. Discards Codex's adversarial perspective exactly when it would most catch a confirmation-biased Claude approval. Parallel dispatch is the cheaper-to-implement and stronger-in-verdict choice.
- **SHA-pin `codex-cli`.** Rejected. The Codex plugin is installed operator-side via `/plugin install codex@openai-codex` and a SHA-pin lives in the plugin marketplace's manifest, not in this repo. Pinning here would create a false-coupling between Mirepoix's PR cadence and OpenAI's plugin release cadence. The runbook documents a soft floor (`codex-cli >= 0.130.0`); below that, the orchestrator treats Codex as unavailable.
- **Inherit Mirepoix-build's Codex-default onto Mirepoix-secure.** Rejected. ADR-010's deny-all-egress posture is the architectural claim of the locked host; inheriting Codex-default would either silently break the gate (Codex calls fail mid-REVIEW from inside the lockdown) or implicitly weaken the firewall (adding OpenAI endpoints to the allowlist without a superseding ADR). Commitment (4) makes Codex-disabled the explicit default on Mirepoix-secure; re-enabling requires a superseding ADR that weakens ADR-010, not a runbook tweak. The bastion-tunnel alternative (routing Codex via the side-by-side bastion) is also rejected here because ADR-010 defines the bastion as a narrow operator tunnel host, not a credential or token relay; turning it into a Codex tunnel would expand its blast radius and is out of scope for this ADR.

## Implementation notes

- **CLAUDE.md is the convention vehicle.** The new `## Multi-agent face-off (Codex as teammate)` section between *Conventions* and *Deployment venues* documents the four commitments for any Claude Code session that loads CLAUDE.md as project context. The on-loop plugin's source is unchanged.
- **No `packages/**` changes.** This ADR is process-level, not harness-level. `@mirepoix/ai` does not gain a Codex provider slot; `@mirepoix/coding` does not gain a `codex` tool. Sub-phase E or later will revisit whether the harness drives Codex programmatically.
- **No cross-repo plugin patch in this PR.** A future cross-repo PR may add the XML-block convention to the on-loop plugin's `architect.md` template so non-Mirepoix consumers also benefit. That work is out of scope here.
- **`command -v codex` is operator shell, not harness bash.** ADR-002 forbids bash allowlists, cwd guards, and path filters inside the harness; the Codex availability probe (`command -v codex` or `/codex:status`) runs in the operator's Claude Code session before/around `/on-loop`, not inside `@mirepoix/coding`'s bash tool. The distinction prevents reviewers from raising a false ADR-002 conflict.
- **ADR-010's `attestation` field is orthogonal.** Mirepoix-secure deployments under ADR-010 declare `attestation: firewall-and-iam` (or stronger) regardless of Codex availability. The sub-phase E carve-out flagged in Consequences may add a Codex-presence dimension to bundle manifests, but ADR-013 does not commit to one.
- **Dogfood.** This PR's REVIEW phase is the first live exercise of the design. The orchestrator dispatches `on-loop:review` (Claude) AND `/codex:adversarial-review` in parallel; both verdicts go into the PR description; either blocks merge. Subsequent `/on-loop` runs inherit the convention from CLAUDE.md without explicit per-PR dogfood instructions.

## Known gaps (v1)

This ADR ships the design at the convention layer. Several enforcement and audit surfaces are deliberately deferred to follow-up sub-phases; v1 is honest about what is **not** code-enforced today.

- **No `MirepoixEvent` arm for Codex verdicts (CRITICAL gap vs. ADR-005).** ADR-005's reconstructability invariant says the JSONL session log records prompts, responses, tool calls, and lifecycle events as the source of truth. v1 records Codex prompts, responses, verdicts, costs, touched files, and stderr only in the PR body and `changes.log` — there is no `codex:*` arm in `MirepoixEvent`. A follow-up sub-phase will land `codex:verdict`, `codex:rescue-start`, `codex:rescue-end`, and `codex:unavailable` event arms in `packages/core/src/events.ts` plus orchestrator-side hooks to emit them. Tracked: [#10 — Land MirepoixEvent codex:* event arms for JSONL audit envelope](https://github.com/UlyssesModel/kavara-mirepoix-internal/issues/10). Until then, audit reconstructability of Codex activity relies on the PR body + `changes.log` pair, not the JSONL session log.
- **Default-on is convention-only, not code-enforced (HIGH).** CLAUDE.md is the load-bearing mechanism that propagates the "every REVIEW dispatches Claude + Codex" rule. The on-loop plugin source is not modified in this PR (cross-repo, out of scope). A future cross-repo PR to `openai/on-loop` will gate REVIEW dispatch on the convention (hard-fail the REVIEW phase if CLAUDE.md declares default-on but Codex was not dispatched, and emit a `review:codex-skip` event). Until that PR lands, operators in non-Mirepoix repos relying on this convention have only the CLAUDE.md honor-system enforcement.
- **Partial outage modes collapse to binary unavailable (MEDIUM).** v1 names three Codex-unavailable conditions: not-installed, not-authenticated, version-incompatible. Real-world failure modes also include job-queued-forever, mid-run rate limit, mid-run auth expiry, malformed structured output, partial stdout, cancelled-from-outside, network timeout, lost background result, and "Codex wrote files then exited nonzero". A follow-up sub-phase will land a state machine + timeout policy and add per-mode `[codex-unavailable] <subreason>` markers. Until then, the orchestrator treats every non-success as the same binary "unavailable" path.
- **XML-block contract has no validator (MEDIUM).** v1 commits the architect to XML-block section style (commitment 3) but does not ship a schema, linter, or downstream consumer that mechanically enforces the tags. Tags risk becoming cargo-cult style rather than load-bearing structure. A follow-up sub-phase will add a schema for required sections (`<task>`, `<structured_output_contract>`, etc.), a markdown-lint rule for tag presence, and documented downstream consumers (coding, testing, security, reviewer agents) that read specific tag contents.
- **Prompt-cost and latency budgets are asserted, not measured (MEDIUM).** v1 claims the parallel-verdict cost is small (Consequences §1). Success criteria currently require dispatch evidence and CI green, not cost metrics. A follow-up sub-phase will add a max review diff size, per-reviewer timeout, token/cost reporting via the JSONL event arms above, and a documented policy for skipping or batching large reviews. Until then, the cost claim is operator-observable but not bounded.
- **`scripts/check-deliverables.sh` lives on a sibling PR (LOW).** The deliverable validation script referenced by the spec's success criteria exists on the `on-loop/harness-deliverable-tracking` branch (PR #9, currently open). This PR was reviewed against `main`, which does not yet have the script. CI ordering between this PR and PR #9 is part of operator coordination — PR #9 should land first; otherwise the success-criteria validation step in this PR's plan must be skipped with a `[check-deliverables-deferred]` marker in `changes.log`.
