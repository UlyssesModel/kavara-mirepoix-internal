# ADR-015: Mirepoix as an ACP server — methodology stack as v0.2.0 distribution surface

Status: Accepted
Date: 2026-05-19
Deciders: John Edge (CTO)
Supersedes: none — extends ADRs 001 (minimal core), 003 (extension model), 005 (audit trail), 010 (Mirepoix-secure deny-all-egress), 011 (attestation enforcement), 012 (two-venue deployment), 013 (codex-as-teammate), and 014 (DDD framework). No prior ADR is invalidated; ADR-015 commits the public-surface shape that v0.2.0 will productize.

## Context

Mirepoix v0.1.x is now operational. The harness packages (`@mirepoix/ai`, `@mirepoix/core`, `@mirepoix/coding`, `@mirepoix/cli`) ship; the methodology stack (mise-en-place behavioral contract, the eight operating modes, on-loop's SDLC pipeline, the grilling skills, the multi-agent face-off pattern per ADR-013, the JSONL audit trail per ADR-005) is operational; the venue split (Mirepoix-build vs. Mirepoix-secure per ADR-012) is documented; the bounded-context framework per ADR-014 is in place. The Scotty TUI spike (v0.1.1, Phase Scotty-A, shipped 2026-05-18 as `UlyssesModel/scotty` v0.1.0) validated Path D end-to-end: an external ACP client consuming an external ACP server (Goose's `goose acp` subprocess) over Bun + Ink + TypeScript with full protocol integration — `initialize`, `session/new`, `session/prompt`, streamed `AgentMessageChunk` / `AgentThoughtChunk` / `ToolCall` notifications, and `session/cancel` clean shutdown.

The architectural-strategic question that has been latent since the v0.1.x scope-cut is: **what shape does Mirepoix Co take when it productizes for v0.2.0?** The implicit assumption in early conversations was that v0.2.0 ships as a SaaS managed service with a custom REST or WebSocket protocol between a Mirepoix-built web frontend and a multi-tenant backend that wraps the existing harness. That assumption was never written down because no ADR forced the question into the open. The Scotty spike forced it: building a TUI on top of Goose's ACP server made the dual-direction property of ACP visible — if Mirepoix can be an ACP *client* of Goose, then symmetrically Mirepoix can be an ACP *server* that other clients consume. That symmetry is not architecturally novel (ACP is designed exactly for this decoupling), but the strategic implication for Mirepoix Co's distribution model is load-bearing enough to commit before v0.2.0 scope is locked.

ACP (Agent Client Protocol) is governed by the Agentic AI Foundation (AAIF) under the Linux Foundation, Apache 2.0 licensed, with a public specification and a growing registry of conformant clients and servers (Zed's agent panel, JetBrains AI assist, multiple VS Code extensions, Goose's own TUI, and now Scotty). The protocol is stable enough for production use: JSON-RPC 2.0 over stdio or HTTP/WebSocket, standard methods (`initialize`, `session/new`, `session/load`, `session/prompt`, `session/cancel`), standard notifications (`AgentMessageChunk`, `AgentThoughtChunk`, `ToolCall`, `ToolCallUpdate`, `request_permission`), and conformance test suites. The protocol is what LSP became for IDE language tooling — a decoupling layer between UI and runtime that becomes the substrate of an ecosystem.

The empirical anchor for committing to ACP as Mirepoix's public surface is six PRs deep on the multi-agent face-off ledger across three product surfaces (kavara-mirepoix-internal PRs #17 and #18 for the Mirepoix harness itself; UlyssesModel/kirk-stub-realistic PRs #2 and #3 for the Kirk pipeline; UlyssesModel/scotty PR #2 for the Scotty TUI) — caught 7+ pre-merge defects that single-reviewer flow would have shipped. The methodology stack is no longer speculative tooling; it has independently validated itself against Rust, Python, and TypeScript codebases under different operational constraints. The methodology is the product. The protocol decision is about how to expose it.

This ADR commits before v0.2.0 spec work begins so that v0.2.0's architecture lands as an explicit ACP-server design rather than a retrofit from a custom protocol that would otherwise be the path of least resistance.

## Decision

ADR-015 makes seven architectural commitments.

The first commitment is that **Mirepoix v0.2.0 ships as an ACP server**. A new package `@mirepoix/acp-server` joins the Harness bounded context (per ADR-014) and exports an ACP-conformant server interface implemented on top of `@mirepoix/core`'s existing agent loop, the typed `MirepoixEvent` bus per ADR-005, the tool surface per ADR-002, and the provider abstraction per ADR-008. The server speaks JSON-RPC 2.0 over stdio (for local-client use cases like Scotty and CI integrations) and over HTTP/WebSocket (for hosted Mirepoix Cloud and remote-client use cases). Custom protocol designs (proprietary REST APIs, WebSocket-with-Mirepoix-specific-message-shapes, GraphQL surfaces) are explicitly rejected — the ACP standard supplies the contract; Mirepoix's value is the methodology stack behind the contract, not the contract itself.

The second commitment is that **the methodology stack maps onto the ACP session lifecycle as the canonical translation**. Specifically:

- **`initialize`** bootstraps a mise-en-place mode for the session. The operator's client supplies the mode in the initialize parameters (`{mode: "build"}`, `{mode: "harden"}`, etc., per the eight modes named in the mise-en-place skill); absent an explicit mode, the server elicits one back through ACP's standard request-response surface before accepting `session/new`. The mode determines which agent behavioral-contract addenda are loaded for the session.
- **`session/new`** allocates a new on-loop pipeline session: assigns a JSONL session log per ADR-005's reconstructability invariant, initializes the bounded-context registry (Harness, Deployment, Tooling, Pipeline, plus the per-tenant context introduced by commitment four below), and returns the `sessionId` that the client uses for subsequent calls.
- **`session/prompt`** is the operator turn. The server routes the prompt through the active mise-en-place mode and the active on-loop phase (architect / coding / testing / security / docs / reviewer / build / git), invokes Claude (or whichever provider is configured per ADR-008), and emits streaming notifications back to the client.
- **`session/cancel`** triggers a graceful cancellation that preserves the JSONL audit trail: the server emits a `cancel:requested` event, waits up to a configurable budget for in-flight tool calls to drain, then emits `session:closed` with the cancellation reason. Mid-tool-call kill is supported but discouraged — the audit trail invariant requires that the JSONL log contain enough information to reconstruct what happened, including partial tool outputs.
- **`session/load`** resumes from a prior session's JSONL log. The server reconstructs the session state from the log per ADR-005's reconstructability invariant, rehydrates the on-loop phase context, and accepts subsequent `session/prompt` calls as if the session had never been suspended.
- **`AgentMessageChunk`**, **`AgentThoughtChunk`**, **`ToolCall`**, **`ToolCallUpdate`**, **`request_permission`** notifications are emitted as the on-loop phase progresses. The mapping from `MirepoixEvent` arms (per ADR-005) to ACP notifications is documented in `@mirepoix/acp-server/EVENTS.md` and reviewed every time `MirepoixEvent` gains a new arm.

This mapping is the Ubiquitous Language (per ADR-014's first commitment) for the Service-Harness boundary. The translation table lives in the new `Service` bounded context's `CONTEXT.md` (per commitment four below).

The third commitment is that **the ACP-server-shape is the primary distribution surface for v0.2.0 and beyond**. Mirepoix Co's commercial offering centers on hosting the ACP server (Mirepoix Cloud) and on the methodology stack that the server exposes — not on a particular UI. Operators consume Mirepoix Cloud from whichever ACP-compatible client they prefer: Zed's agent panel, JetBrains AI assist, the various VS Code extensions, Scotty (for Mirepoix-secure use cases), Goose's TUI (configured to point at Mirepoix-as-backend rather than Goose's own backend), CI/CD pipelines making programmatic ACP calls, and the Mirepoix Cloud web frontend (which is itself an ACP client wrapping a hosted ACP server). The UI is replaceable; the methodology assembly is the moat.

The fourth commitment is that **a new bounded context — `Service` — is introduced** to own the multi-tenant, hosting, and commercialization concerns. The Service context covers tenant identity and routing, authentication and authorization (initially v0.2.0 single-tenant via operator-owned API keys; v0.2.1 multi-tenant via OIDC/SAML/passkey), metering and billing (per-session counts, per-methodology-feature gating, per-tool-call cost attribution), per-tenant resource quotas, per-tenant data residency policies, and the API surface beyond ACP itself (admin endpoints, billing webhooks, audit-log export). The Service context is a Supporting Subdomain per the ADR-014 Subdomain-classification axis — necessary infrastructure that supports the Core Domain (Harness), with operational rigor and well-defined boundary contracts, but where invention happens only where the commercial proposition demands it (e.g., metering shape) and not where commodity solutions exist (e.g., OIDC). The `CONTEXT-MAP.md` at the repo root is updated to declare the fifth context and its boundary with Harness; the new `docs/service/CONTEXT.md` ships as a skeleton ready to be populated by grilling sessions.

The fifth commitment is that **authentication, authorization, and multi-tenant isolation are deferred to v0.2.1 but the boundary is architected from v0.2.0 day one**. v0.2.0 ships single-tenant: one operator, operator-owned data, operator-owned API keys for the provider (Claude, Ollama, etc.) supplied directly to the server. v0.2.0 is sufficient to validate the ACP-server shape end-to-end and to prove the methodology stack is exposable. v0.2.1 introduces tenant identity, SSO (OIDC and SAML; passkeys via WebAuthn), per-tenant key management (BYOK and managed-keys options), and the per-tenant isolation enforcement at the ACP-server layer. The architectural commitment in v0.2.0 is that the ACP server is structured such that v0.2.1's auth/tenancy plug-in does not require rewriting the server shape — the session-creation path, the bounded-context registry, and the audit-trail emission must already be tenant-aware (with `tenant: "solo-operator"` as the v0.2.0 placeholder) and not be retrofitted onto a single-tenant assumption. This is the deferred-but-shaped pattern: defer the implementation, commit the shape.

The sixth commitment is that **the ADR-010 / ADR-012 venue split composes onto the ACP-server architecture as follows**. Mirepoix-build venue consumes hosted Mirepoix Cloud (the managed ACP server) or runs the ACP server locally (`@mirepoix/acp-server` invoked via stdio from any ACP client); both modes are supported and which one an operator uses is an operational choice. Mirepoix-secure venue (deny-all-egress per ADR-010, on `scotty-gpu`) **cannot** consume hosted Mirepoix Cloud — egress to the managed service violates the firewall posture. Mirepoix-secure deployments run a local `@mirepoix/acp-server` instance on the locked host, configured to use harness-local providers (Ollama serving Qwen2.5-Coder per the ADR-008 substrate-aware-serving design), with Scotty (the ACP client validated in v0.1.1) as the operator-facing UI. The local ACP server's audit trail (per ADR-005) lives on the locked host's local storage and is exfiltrated through the bastion's narrow tunnel (per ADR-010) at operator discretion. This composition makes the venue split into the *deployment shape* of the ACP server, not into two independent architectures: same `@mirepoix/acp-server` package, two operational modes, one continuous Ubiquitous Language across both venues.

The seventh commitment is that **the multi-agent face-off pattern per ADR-013 lifts to a property of the ACP-server architecture rather than remaining a property of the on-loop plugin**. Any ACP client consuming Mirepoix-as-ACP-server gets the face-off discipline automatically — when the server's REVIEW phase runs, it dispatches the Claude and Codex teammates in parallel per ADR-013's first commitment, posts both verdicts into the session's audit trail and emits them as `request_permission` notifications back to the client (where the client renders both verdicts and the operator decides on the gate). Operators using Zed, JetBrains, VS Code, or Mirepoix Cloud's web frontend get the face-off discipline as a built-in property of the Mirepoix backend; they do not have to install Codex separately or wire up the face-off pattern themselves. The Codex teammate's availability on the server side follows ADR-013's commitments four and five (Mirepoix-build default-on; Mirepoix-secure default-off; degraded fallback to Claude-only with the `[codex-unavailable]` marker). The audit-trail invariant per ADR-005 is preserved: the JSONL log records both teammates' verdicts verbatim regardless of which client renders them.

## Consequences

The first consequence is that **UI becomes pluggable across Mirepoix's distribution surface**. Mirepoix Co's commercial proposition is no longer constrained by which UIs Kavara builds first-party — it is constrained by which UIs the ACP ecosystem already supports plus whichever first-party clients Kavara chooses to invest in. Zed, JetBrains, VS Code, and Goose's TUI users can already consume Mirepoix Cloud once it ships, without Kavara writing any client code for those editors. Scotty handles Mirepoix-secure operator UX. The Mirepoix Cloud web frontend (v0.2.0 launch surface) handles the SaaS-default UX. First-party client investments for Zed/JetBrains/VS Code become a v0.3.0+ question of branding leverage rather than a distribution prerequisite.

The second consequence is that **distribution leverage compounds with the ACP ecosystem rather than with Kavara's internal client-build velocity**. Every new ACP-compatible client that emerges in the ecosystem automatically gains Mirepoix as a backend option. Kavara does not have to build N integrations across N editor vendors; Kavara builds one ACP server and inherits the ecosystem. This is the same network-effect leverage that LSP gave to language tooling vendors — the protocol's adoption is the moat-amplifier.

The third consequence is that **Mirepoix Co's positioning thesis becomes literally true at the architectural level**. The earlier positioning ("Mirepoix Co sells the methodology layer for agentic coding, not model access") was a marketing frame that the architecture had to grow into. Under ADR-015, the product *is* the methodology server; the UI surface is whichever client the operator prefers; the model access is a configuration parameter (provider abstraction per ADR-008). Marketing copy no longer needs to explain "we're the methodology layer, not the model" — the architecture itself says it.

The fourth consequence is that **v0.2.0 engineering surface narrows materially**. Under the implicit-custom-protocol assumption, v0.2.0 would have required designing and implementing a Mirepoix-specific REST or WebSocket protocol between the web frontend and the backend — a multi-quarter task with non-trivial security, versioning, and tooling concerns. Under ADR-015, the contract is ACP, which already exists. v0.2.0's engineering surface narrows to: implement `@mirepoix/acp-server`, ship the Service bounded context with single-tenant config and v0.2.0 metering hooks, build (or adapt) the Mirepoix Cloud web frontend as an ACP client, and integrate with the conformance test suite. Multi-quarter to single-quarter compression on the protocol-layer work.

The fifth consequence is that **the ADR-005 audit trail becomes the source of truth for commercial concerns**. Per-session metering, per-methodology-feature gating, per-tool-call cost attribution, and per-tenant audit-log export all read from the JSONL session log. The audit trail invariant gains commercial gravity in addition to its existing reconstructability gravity and ADR-011's attestation gravity — operational changes that compromise audit-trail completeness now risk breaking billing accuracy as well as breaking forensic reconstruction and attestation reconstruction. This raises the bar for any future change that touches the audit-trail format (the bar is already high per ADR-005; ADR-011's attestation enforcement and ADR-015's commercial use make it functionally a one-way door).

The sixth consequence is that **the Codex face-off pattern per ADR-013 becomes a marketed value proposition for Mirepoix Cloud customers** rather than an internal Kavara discipline. The six PRs of empirical validation (Mirepoix harness PRs #17/#18; kirk-stub-realistic PRs #2/#3; Scotty PR #2; with PR #5's post-merge grilling as the related-but-distinct anchor) become customer-facing proof points. Customers consuming Mirepoix Cloud through any ACP client get multi-agent review discipline automatically, without having to install Codex or wire up the face-off themselves. This is the kind of capability that distinguishes a managed-methodology offering from a self-host-the-harness-yourself open-source play.

The seventh consequence is that **a small competitive vulnerability is now visible**. Because ACP is an open protocol, a competitor could in principle implement their own methodology server speaking ACP and compete on the methodology dimension directly. Mirepoix's defense is the six-month methodology assembly itself — the architectural opinions captured in ADRs 001–015, the bounded-context discipline per ADR-014, the audit-trail invariant per ADR-005, the venue-split rigor per ADR-010 and ADR-012, the multi-agent face-off pattern per ADR-013, the eight-mode behavioral contract from mise-en-place, the on-loop SDLC pipeline discipline, the grilling skills. Cloning the protocol is trivial; cloning the methodology assembly is the actual moat. This is the same competitive posture as LSP vendors who sell language services — the protocol is open, the implementation expertise is the differentiator. ADR-015 accepts this posture knowingly.

The eighth consequence is that **a fifth bounded context joins the architecture without superseding the four named in ADR-014**. ADR-014 named Harness, Deployment, Tooling, Pipeline. ADR-015 adds Service. The CONTEXT-MAP.md gains an entry; a new `docs/service/CONTEXT.md` ships as a skeleton; the first grilling session against the Service context populates the canonical terms (tenant, identity, metering, quota, BYOK, etc.). The DDD framework per ADR-014 made this addition legible without ceremony — extending the bounded-context registry is exactly what the framework is designed to accommodate.

## Alternatives considered

We considered shipping v0.2.0 with a **custom REST/WebSocket protocol** between a Mirepoix-built web frontend and the harness backend. Rejected. The protocol surface alone would consume multi-quarter engineering capacity (versioning, auth, streaming semantics, tooling, conformance tests) that ACP already provides. The custom protocol also forecloses ecosystem distribution — operators using Zed, JetBrains, VS Code, or Goose's TUI would be locked out of Mirepoix Cloud entirely unless Kavara built first-party clients for each editor, which is a non-starter at v0.2.0 scope. The custom protocol path produces a slower-to-ship, less-distributed, more-engineering-heavy v0.2.0 with no compensating commercial benefit.

We considered shipping v0.2.0 as a **license-only model** — selling the methodology stack as a self-hosted software license without a managed-service offering. Rejected. The license-only path discards the SaaS-side commercial leverage (recurring revenue, usage-based pricing, multi-tenant operational economies, audit-log-as-billing-source). Self-hosted-only also raises the adoption bar materially: every customer becomes an infrastructure customer; operational support burden compounds; the methodology assembly's value is gated behind customer infrastructure competency. Mirepoix Co's commercial model is plausibly *both* license (for Mirepoix-secure and on-prem customers) *and* managed service (for Mirepoix-build and Mirepoix Cloud customers), but ADR-015 commits to the managed-service shape as v0.2.0's primary surface; license offerings are out of scope here and may justify their own ADR.

We considered **building on someone else's ACP server** (e.g., extending Goose's ACP server with Mirepoix's methodology overlay). Rejected. The methodology stack is Mirepoix's differentiator; building on Goose makes Goose the platform and Mirepoix a plugin to it, inverting the strategic positioning. Goose is an excellent client (Scotty consumes it as one) and an excellent reference implementation of an ACP server, but Mirepoix's commercial proposition is not "Goose plus some plugins" — it is the methodology assembly as the primary offering. The ACP protocol's value is precisely that we do not have to build on Goose; we can build a peer server that interoperates with the same clients.

We considered **waiting until ACP matures further** before committing v0.2.0's architecture to it. Rejected. ACP is now governed by AAIF/Linux Foundation, is on a documented stable release cadence, has a conformance test suite, has a growing registry of clients and servers in production (Zed, JetBrains, VS Code, Goose, Scotty), and has the Apache 2.0 license posture that the Kavara organization is comfortable taking dependencies on. Waiting buys protocol-stability margin we do not need and costs Mirepoix Co the v0.2.0 distribution leverage that ACP provides today. The opportunity cost of waiting outweighs the marginal protocol-stability gain.

We considered **introducing the Service bounded context as part of ADR-014** rather than waiting for ADR-015. Rejected at the time ADR-014 was drafted because no architectural commitment to a service shape existed yet — naming a context before there was a decision about what lived inside it would have been speculative DDD application, which ADR-014 explicitly rejects. ADR-015's commitment to the ACP server shape is precisely the architectural decision that justifies introducing the Service context now. The DDD framework's discipline is exactly that contexts emerge when the commitments warrant them; they do not get reserved in advance.

We considered **shipping v0.2.0 multi-tenant from day one** rather than deferring tenancy to v0.2.1. Rejected. Multi-tenant auth, identity, isolation enforcement, and per-tenant billing are non-trivial engineering surfaces that would extend v0.2.0's scope by multiple quarters. v0.2.0's load-bearing job is to prove the ACP server shape end-to-end and validate the methodology-as-product thesis with real operators. Single-tenant is sufficient for that; multi-tenant is the productionization step that v0.2.1 handles. Commitment five's deferred-but-shaped pattern explicitly protects against the retrofit risk by requiring v0.2.0's session creation, bounded-context registry, and audit-trail emission to be tenant-aware from day one even when only one tenant exists.

We considered **emitting `MirepoixEvent` arms identical to the existing ones for ACP-mapped events** rather than introducing `acp:*` arms. Rejected. The Ubiquitous Language per ADR-014 makes this a real cost — using identical names across the Harness-internal and Service-external surfaces would obscure which events are protocol-observable (and therefore part of the customer-facing contract) versus internal to the harness. The `acp:*` arm prefix marks events as customer-facing surface, which has security, versioning, and documentation implications that internal events do not have. Naming makes the boundary legible.

## Implementation notes

This ADR commits the architecture but does not itself ship the v0.2.0 implementation. The implementation ships in a sequence of sub-phases, each tracked in `IMPLEMENTATION-PLAN.md` and gated by the on-loop SDLC pipeline.

The first implementation deliverable is `@mirepoix/acp-server` as a new package in the Harness bounded context. Initial scope: stdio transport (HTTP/WebSocket deferred to the second sub-phase), the seven ACP methods named in commitment two, the notification mapping table from `MirepoixEvent` arms to ACP notifications, the EVENTS.md contract document, and conformance against the AAIF ACP test suite. The package consumes `@mirepoix/core`'s existing agent loop and `@mirepoix/coding`'s tool surface without internal-API leakage — the ACP server is a wrapper, not a fork.

The second implementation deliverable is the HTTP/WebSocket transport for `@mirepoix/acp-server`. Required for hosted Mirepoix Cloud; the stdio transport is sufficient for the local-client use cases (Scotty, CI integrations, local IDE plugins) but Mirepoix Cloud requires a network-reachable transport. The HTTP/WebSocket transport must conform to the ACP spec's transport binding and pass the conformance suite.

The third implementation deliverable is the Service bounded context skeleton. CONTEXT-MAP.md is updated to declare the fifth context; `docs/service/CONTEXT.md` ships with placeholders for tenant, identity, metering, quota, BYOK, residency, etc. The first grilling session against the Service context populates the canonical terms.

The fourth implementation deliverable is v0.2.0 metering hooks. `@mirepoix/acp-server` emits metering events to the Service context's metering sink (initially a local-file sink for the single-tenant v0.2.0 deployment; the hosted-multi-tenant sink lands in v0.2.1). Specific `acp:*` `MirepoixEvent` arms: `acp:session-init`, `acp:session-prompt`, `acp:session-cancel`, `acp:session-load`, `acp:tool-call`, `acp:tool-call-update`, `acp:request-permission`, `acp:metering-tick`. The mapping to billable units is documented in the Service `CONTEXT.md`.

The fifth implementation deliverable is the Mirepoix Cloud web frontend as an ACP client. The frontend talks to a hosted `@mirepoix/acp-server` over the HTTP/WebSocket transport. Same protocol that any other ACP client speaks; no custom integration. UI design and feature scope are out of scope for this ADR.

The sixth implementation deliverable is the ACP registry publication. Once v0.2.0 is stable, Mirepoix-as-ACP-server is submitted to the public AAIF ACP registry (https://github.com/agentclientprotocol/registry). Distribution becomes "any ACP client can find and connect."

A subsequent ADR (ADR-016 or later) will revisit the deferred concerns: multi-tenant auth and SSO (commitment five's v0.2.1 territory), per-tenant data residency policy, the license-vs-managed-service offering split, the pricing-tier definition (specific tier names, specific gating rules), and the v0.3.0+ first-party Zed/JetBrains/VS Code client investment decisions. ADR-015 commits the protocol-and-distribution architecture; subsequent ADRs commit the productization specifics.

The empirical validation pattern from PR #5 / PR #18 / PR #2 (kirk) / PR #2 (scotty) / PR #3 (kirk Option B) — the multi-agent face-off catching real pre-merge defects — applies to the v0.2.0 implementation work. Every `@mirepoix/acp-server` sub-phase ships through on-loop with the Codex teammate enabled per ADR-013's first commitment. The methodology stack is being used to ship the methodology stack's productized surface — dogfood at the architectural level.

## Known gaps (v1)

This ADR ships the architectural commitment at the framework layer. Several productization surfaces are deferred to follow-up sub-phases and follow-up ADRs; v1 is honest about what is **not** committed today.

- **Multi-tenant auth and SSO are committed in shape, not in specification (HIGH).** Commitment five names the deferred-but-shaped pattern but does not specify which OIDC providers, which SAML semantics, which passkey flows, or which BYOK key-management scheme v0.2.1 will adopt. The v0.2.0 implementation must architect the boundary such that v0.2.1's choices plug in cleanly; the v0.2.1 ADR will lock the specifics. Until v0.2.1 lands, the only supported tenancy mode is single-tenant with operator-owned keys.
- **Pricing model is committed at the architectural-hook layer, not at the tier-definition layer (HIGH).** ADR-015 commits per-session metering and per-methodology-feature gating as the architectural primitives. Specific tier names, specific price points, specific gating rules ("does mise-en-place mode `harden` count against a higher tier than `build`?") are out of scope; they are commercial decisions that depend on customer development and pricing research. A subsequent ADR (probably ADR-017 or later) will lock the tier definitions when the commercial framing is ready.
- **ACP protocol version pinning is not committed (MEDIUM).** The AAIF ACP spec is on a documented release cadence with semantic versioning. ADR-015 commits Mirepoix to ACP-conformance generally but does not pin a specific protocol version. A subsequent sub-phase will land an explicit version-floor (`acp >= X.Y.Z`) in `@mirepoix/acp-server`'s conformance manifest, the version-bump policy (semantic-versioning-aware upgrades), and the deprecation policy for clients on older protocol versions. Until then, the package will conform to the latest stable spec at the time of release.
- **Mirepoix Cloud's web frontend is named, not designed (MEDIUM).** Commitment three names "the Mirepoix Cloud web frontend (v0.2.0 launch surface)" but does not specify the UI, the framework choice (React vs. Svelte vs. SolidJS), the rendering shape (server-side vs. SPA), or the feature scope. UI design is out of scope for ADR-015; a subsequent design ADR or PRD-shaped document will lock it. v0.2.0's load-bearing job is the protocol; the UI is a deliverable that depends on the protocol but does not architecturally constrain it.
- **First-party Zed/JetBrains/VS Code clients are v0.3.0 territory, with no investment decision (LOW).** Commitment three's claim that "operators consume Mirepoix Cloud from whichever ACP-compatible client they prefer" works in v0.2.0 because Zed, JetBrains, and VS Code already have ACP clients. Whether Kavara invests in first-party Mirepoix-branded clients for those editors in v0.3.0+ is a separate branding-leverage decision that depends on commercial signal — if Mirepoix Cloud customers strongly prefer a Mirepoix-native UX over the third-party-editor UX, Kavara may invest; otherwise the third-party clients are sufficient. ADR-015 does not commit either way.
- **License-vs-managed-service offering split is not committed (LOW).** ADR-015 commits the managed-service shape (Mirepoix Cloud) as v0.2.0's primary surface but leaves the license-only offering open for Mirepoix-secure and on-prem customers. Whether Mirepoix Co offers both, only managed, or eventually only license is a commercial decision that depends on customer development. A subsequent ADR may commit the offering split when the commercial framing is ready.
- **`acp:*` `MirepoixEvent` arm specifications are named, not landed (LOW).** Implementation note four names eight `acp:*` event arms but does not specify their payload shapes. The shapes will land in `packages/core/src/events.ts` as part of the `@mirepoix/acp-server` sub-phase, reviewed against ADR-005's reconstructability invariant. Until that sub-phase lands, the JSONL session log records ACP-mapped events using the existing internal arms (e.g., `phase:review-start` rather than `acp:session-prompt`); a downstream pass will migrate the audit trail to the `acp:*` shape with a documented backwards-compatibility window.
