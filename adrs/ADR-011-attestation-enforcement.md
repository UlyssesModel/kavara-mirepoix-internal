# ADR-011: Attestation enforcement — promoting the `attestation` field from observational to enforced

Status: Accepted
Date: 2026-05-19
Deciders: John Edge (CTO)
Supersedes: none — extends ADR-010 (Mirepoix-secure pilot) by promoting one of the forward-referenced refinements; harmonizes with the parallel Kirk-real work documented in Confluence PE/107970562 (ADR-0001 — Two-Plane Build Topology), PE/110919681 (ADR-0002 — Image Categorisation Rubric), and PE/111017985 (ADR-0003 — Customer-Verifiable IP Attestation Chain). Forward-compatible with ADR-015 (Mirepoix as ACP server).

## Context

ADR-010 shipped Mirepoix-secure as a deny-all-egress posture on `scotty-gpu` with a Phase-a validated baseline, and named four future refinements in its forward-reference section: promoting the `attestation` field from observational to enforced; adding Mirepoix-restricted as a less-strict peer posture; committing to a Phase-b procurement timeline; and extending the posture to Azure TDX or AWS AMD SEV-SNP substrates. The `attestation` field appears in Mirepoix-secure's bundle manifest as a declarative posture statement (`attestation: firewall-and-iam`, `attestation: tdx-measured`, etc.) — operators write it to claim a compliance level; the field is read by audit tooling and rendered in the audit-grade compliance documentation, but nothing in the publication pipeline, the runtime, or the operational tooling actually verifies that the claimed attestation level is achieved before the bundle ships to customers. Observational, in ADR-010's language.

The Kirk-real team's parallel work has developed the mechanical attestation chain that observational mode lacks. Confluence PE/107970562 (ADR-0001 — Two-Plane Build Topology) committed Kirk-real images to build only on TDX-attested appliance hosts and named the Public Plane / IP Plane boundary as the architectural gate. PE/110919681 (ADR-0002 — Image Categorisation Rubric) committed the per-image classification rubric that determines which plane an image lives on. PE/111017985 (ADR-0003 — Customer-Verifiable IP Attestation Chain for Kirk-Real Images) committed the customer-verifiable evidence chain — signed manifests, transparency-log entries, and the customer-side verification primitive that lets a customer independently confirm that a Kirk-real image was built on an attested appliance host. That work was driven by Kirk-real's commercial requirement (customers buying Kirk-confidential workloads will not consume images they cannot independently verify) and has produced a working attestation chain at the operational layer.

The architectural debt this exposes is that **Mirepoix's `attestation` field is currently a marketing claim, not an enforced property**. Mirepoix-secure bundles can ship with an `attestation: tdx-measured` declaration regardless of whether the build host actually produced a TDX measurement quote, whether the quote was verified, or whether the bundle's contents match the measured baseline. A customer reading the manifest sees the claim; the customer's compliance officer reads the claim; the Mirepoix Cloud sales pitch quotes the claim — but the architecture does not back the claim. This is exactly the kind of architectural-claim-outpacing-architectural-implementation that ADR-014's DDD framework explicitly calls out as vocabulary drift becoming a defect, and is exactly the kind of audit-grade-compliance vulnerability that ADR-015's commercial proposition (Mirepoix Cloud as the managed-methodology offering with audit-grade traceability) cannot tolerate.

This ADR commits before v0.2.0 productization work on ADR-015 begins so that the v0.2.0 surface ships with attestation enforced rather than observational. It closes one of ADR-010's four reserved-territory threads; the other three (Mirepoix-restricted peer posture, Phase-b procurement timeline, multi-substrate posture) remain reserved for future ADRs and may justify their own slots when commercial or operational signal warrants them.

## Decision

ADR-011 makes five architectural commitments.

The first commitment is that **the `attestation` field is promoted from observational to enforced at bundle publication**. The Mirepoix bundle-publication pipeline gains a verification gate that runs before a bundle can be tagged for release: the gate reads the bundle's `attestation` declaration, executes the corresponding verification procedure for the declared substrate, and either approves the publication or rejects it. Approved bundles ship with a signed attestation manifest as a sibling artifact; rejected bundles do not ship at all. The gate's verdict lands in the JSONL audit log per ADR-005's reconstructability invariant — every publication decision carries an `attestation:verify` event arm capturing the substrate, the measurement quote (or its hash), the verification outcome, and the operator who triggered the publication. There is no observational fallback path; an attestation declaration that fails verification is a publication failure, not a downgrade. The Harness bounded context per ADR-014 gains a new sibling package `@mirepoix/attestation` that owns the verification logic, the manifest format, the substrate-specific verifiers, and the audit-trail emission. The package is dependency-injectable into `@mirepoix/cli`'s publication command in v0.2.0 and into `@mirepoix/acp-server` per ADR-015 when the ACP-server productization lands.

The second commitment is that **GCP TDX is the v1 substrate profile and is the only profile that ships in v0.2.0**. The verification procedure for `attestation: gcp-tdx-measured` consists of: (a) the build host produces an Intel TDX measurement quote covering the build environment and the bundle artifacts; (b) the quote is verified against Intel's TDX attestation service via the standard verification path; (c) the verified quote, the bundle's content hashes, and the operator's GPG signature are bundled into a signed attestation manifest in JSON Web Signature (JWS) format; (d) the manifest is published as a sibling artifact alongside the bundle in the same registry. Other substrates (Azure TDX via Azure Confidential Computing, AWS Nitro Enclaves, AWS AMD SEV-SNP, on-premises bare-metal TDX appliances) are explicitly out of v1 scope; they require their own verifiers, their own attestation chains, and their own customer-side verification surfaces. Per-substrate profile additions are reserved for future ADRs as commercial signal arrives. The v1 profile is sufficient for Mirepoix-secure deployments on `scotty-gpu` (which is GCP-hosted) and for Mirepoix Cloud customers on GCP; multi-cloud Mirepoix Cloud is a v0.3.0+ question.

The third commitment is that **the customer-verifiable evidence chain harmonizes with Confluence ADR-0003's Kirk-real attestation chain**. Customers consuming Mirepoix-secure bundles use the same verification primitive that Kirk-real customers use to verify Kirk-real images: a customer-side CLI (`@mirepoix/verify-attestation`, publicly installable from npm) that reads the signed attestation manifest, re-verifies the measurement quote against Intel's attestation service, re-computes the bundle's content hashes from the local download, and reports the cross-check verdict. The CLI is identical in shape and behavior to the Kirk-real verification tool referenced in Confluence PE/111017985 — same JSON Web Signature format for manifests, same content-hash conventions, same Intel TDX verification path, same exit-code semantics. The vocabulary alignment is intentional per ADR-014's Ubiquitous Language commitment: Kirk-real customers and Mirepoix-secure customers see the same compliance UX even though the underlying products differ. The implementation may share a code library between the two product surfaces (Kirk-real's tool and `@mirepoix/verify-attestation`) once the operational pattern is validated in production; v0.2.0 ships separate-implementations-with-aligned-vocabulary, and a later ADR may commit consolidation if the operational signal warrants it.

The fourth commitment is that **runtime drift detection is named but deferred**. The publication gate (commitment one) verifies that the bundle was built on an attested host with a verified measurement. It does not verify that the running instance of the bundle continues to match the attestation manifest at runtime. Runtime drift detection (binary diff against the manifest's content hashes, periodic re-attestation against Intel's service, alert routing on drift) is a known gap that future operational tooling will close; for v1 the commitment is publication-time enforcement only. Customers who require runtime drift detection are documented as needing the Mirepoix-secure venue (where deny-all-egress + bastion-mediated audit per ADR-010 provides a tighter operational envelope) rather than relying on attestation enforcement alone. This is the deferred-but-shaped pattern in ADR-015's vocabulary — defer the implementation, commit the architectural boundary so future drift-detection tooling has a clean place to plug in.

The fifth commitment is that **the migration path for existing observational bundles is committed at the policy layer, not at the tooling layer**. Bundles published before ADR-011's effective date carry observational `attestation` declarations; they continue to work as before — no retroactive verification, no retroactive rejection, no breaking change for existing customers. Bundles published after ADR-011's effective date must pass the publication gate; the `attestation` field is no longer a free-text declaration but a verified property. The transition is policy-only: there is no migration script, no bundle re-signing pipeline, no retroactive attestation manifest synthesis. The policy date is the day this ADR merges. Customer-side `verify-attestation` runs on pre-ADR-011 bundles will report `attestation: observational (pre-enforcement)` rather than `verified` or `failed`; that's the explicit observational signal in the post-enforcement world. Customers consuming bundles published after the policy date receive the enforced signal automatically. This trades clean migration tooling for clean architectural commitment — operators who want enforced attestation on a pre-ADR-011 bundle must republish it through the new pipeline rather than relying on retroactive metadata patching.

## Consequences

The first consequence is that **the Mirepoix compliance posture gains architectural authority**. Before ADR-011, the `attestation` field was a marketing claim that a sufficiently motivated customer compliance officer could discredit by asking "but where's the verification?" After ADR-011, the field is a verified property with a customer-runnable verification primitive that produces a binary verdict. Marketing copy ("Mirepoix Cloud bundles ship with verified TDX attestation") becomes truthful at the architectural level rather than aspirational at the documentation level. This is the same harmonization pattern ADR-015 commits for its "the methodology is the product" positioning — the architecture has to back the marketing claim, not the other way around.

The second consequence is that **bundle publication latency increases by the TDX verification round-trip**. Intel's TDX attestation service has documented response-time characteristics in the sub-second-to-few-seconds range under normal load; the publication gate adds this round-trip to the publication-time critical path. For low-frequency publications (Mirepoix's typical case — sub-phase releases at human cadence) the latency is operationally invisible. For high-frequency automated publications (which Mirepoix does not currently have) the latency could compound; v1 explicitly does not optimize for this case. The publication pipeline emits a `publication:latency-budget-exceeded` warning if a single publication's attestation-verification round-trip exceeds 30 seconds, surfacing the operational signal before the latency becomes a customer-facing problem.

The third consequence is that **multi-cloud expansion is now architecturally harder than it was before**. Adding Azure TDX or AWS Nitro Enclaves as a substrate profile now requires implementing a per-substrate verifier, validating it against the substrate's attestation service, harmonizing the manifest format across substrates, and shipping the customer-side `verify-attestation` updates that handle each substrate's evidence shape. This is real engineering cost. ADR-011 accepts it knowingly — the alternative (verifying-only-on-GCP-and-claiming-everywhere) would be exactly the marketing-outpacing-architecture failure mode that the ADR exists to close. The cost of doing this properly per-substrate is the price of having an enforced posture at all; teams that want loose multi-cloud should not adopt the enforced posture in the first place. Future per-substrate ADRs (ADR-018 or beyond, per ADR-010's reserved-territory pattern) will commit each profile when commercial signal warrants the engineering investment.

The fourth consequence is that **ADR-015's v0.2.0 commercial proposition gains a concrete compliance feature**. The ACP-server architecture committed in ADR-015 includes audit-trail emission per ADR-005 and metering per ADR-015's commitment four. With ADR-011 in place, every Mirepoix Cloud bundle the customer pulls down is a verified-attestation artifact, the customer can independently re-verify the chain, and the audit-trail captures the verification verdict. Mirepoix Cloud's compliance posture page now has three real architectural anchors (audit trail, attestation enforcement, deny-all-egress for Mirepoix-secure customers) rather than two anchors and a marketing claim. The kind of customer who pays a premium for audit-grade compliance — federal, healthcare, financial-services — now sees the architectural property they need.

The fifth consequence is that **the Kirk-real two-plane build topology gains a Mirepoix-side anchor**. Confluence PE/107970562's two-plane topology is an operational architecture in the Kirk product family; it lacked a corresponding architectural anchor in the Mirepoix harness sequence. ADR-011 provides that anchor: the `@mirepoix/attestation` package is the Harness-side counterpart to Kirk-real's appliance-host build pipeline, and the `verify-attestation` customer tool is the Harness-side counterpart to Kirk-real's customer-side verifier. The two ADR sequences (Kirk PE / Mirepoix GitHub) now document the same architectural reality from coherent angles per ADR-014's DDD framing — Kirk's PE sequence describes the substrate side (how attested build hosts are provisioned, what the per-image rubric is, how the chain is signed); Mirepoix's GitHub sequence describes the harness side (how the bundle verifies the attestation, how the audit trail records the verdict, how the customer reproduces the cross-check).

The sixth consequence is that **the ADR-005 audit-trail invariant gains attestation gravity in addition to its existing reconstructability gravity and ADR-015's commercial gravity**. Every publication decision now lands an `attestation:verify` event arm in the JSONL log; that arm becomes the source of truth for compliance audits, for customer dispute resolution, and for the customer-side verification cross-check. Operational changes that touch the audit-trail format now risk breaking attestation reconstruction in addition to billing accuracy (per ADR-015) and forensic reconstruction (per ADR-005). The bar for audit-trail-format changes was already high; with three load-bearing dependencies on it, the bar is now functionally a one-way door.

The seventh consequence is that **operational tooling burden increases by the per-substrate verifier maintenance**. Each substrate Mirepoix supports for enforced attestation requires keeping the substrate's verifier current against the substrate's attestation service evolution — Intel's TDX service has versioned APIs, deprecation cycles, and certificate-rotation events that the Mirepoix verifier must track. In v1 (GCP TDX only) the burden is bounded; as multi-substrate ADRs land, the burden scales linearly with substrate count. ADR-011 accepts this knowingly; the alternative (a single generic verifier across substrates) is architecturally cleaner but operationally fragile because different substrates' attestation services have genuinely different evidence shapes that don't compose under a generic abstraction without leaking the substrate's quirks into the abstraction.

## Alternatives considered

We considered **keeping the `attestation` field observational** and treating ADR-010's forward reference as aspirational rather than load-bearing. Rejected. The observational posture is exactly the architectural-claim-outpacing-architectural-implementation failure mode that ADR-014's DDD discipline calls out, and it directly contradicts ADR-015's commercial proposition (Mirepoix Cloud as the audit-grade managed-methodology offering). The promise made in ADR-010 has been on the books for two weeks and the parallel Kirk-real work has built the operational pattern; the architectural commitment must catch up.

We considered **enforcing only at runtime, not at publication**. Rejected. Runtime enforcement is necessary but not sufficient — if the bundle was built on an unattested host, runtime measurement quotes will not match the manifest because the manifest is wrong-by-construction; the failure surfaces only when a customer pulls the bundle and verifies, which is too late. Publication-time enforcement catches misconfiguration before customer impact. Runtime drift detection (commitment four's deferred-but-shaped pattern) is an additional safeguard, not a substitute.

We considered **enforcing only for Kirk-real bundles, not for all Mirepoix-secure bundles**. Rejected. A two-tier compliance posture inside Mirepoix-secure (enforced for Kirk-real, observational for everything else) would be operationally confusing — the same `scotty-gpu` deployment posture, two different attestation behaviors depending on the workload type. The compliance proposition fragments along workload-classification lines, which is exactly the kind of vocabulary-drift-becoming-architectural-defect that ADR-014 was adopted to prevent. ADR-011 commits enforcement at the Mirepoix-secure level (and at the Mirepoix-build level when bundles declare attestation, which is rare but possible); the workload type is orthogonal to the enforcement boundary.

We considered **deferring attestation enforcement to v0.2.1 alongside multi-tenant auth**. Rejected. Attestation enforcement is architecturally independent of tenancy — single-tenant Mirepoix Cloud customers benefit from attestation enforcement just as much as multi-tenant ones, and the architectural surfaces (verification logic, manifest format, customer-side primitive) do not share design constraints with auth or SSO. Coupling them into v0.2.1 would delay attestation enforcement for no compensating architectural benefit. ADR-011 ships before v0.2.0's productization work begins so that the v0.2.0 surface launches with attestation already enforced.

We considered **building a single generic substrate verifier** that abstracts over GCP TDX, Azure TDX, AWS Nitro Enclaves, AWS AMD SEV-SNP, and on-premises TDX appliances. Rejected. The substrates have genuinely different evidence shapes — Intel TDX produces signed measurement quotes; AWS Nitro Enclaves produce signed attestation documents with different field structure; AMD SEV-SNP produces different again. A generic abstraction that papers over the differences either leaks substrate-specific quirks at the boundary (defeating the abstraction's purpose) or hides substrate-specific failure modes from the verification logic (which is precisely what attestation enforcement exists to prevent). Per-substrate verifiers are the architecturally honest path; the linear scaling cost is the price of substrate-honesty.

We considered **a third-party attestation broker** (e.g., a SaaS service that abstracts over substrates and provides a unified verification API). Rejected at v1 scope. Third-party brokers reduce per-substrate engineering burden at the cost of introducing a dependency on the broker's availability, security posture, and pricing model — none of which Mirepoix Cloud's customers can independently verify, and at least one of which (the broker's security posture) is exactly what the customer was trying to verify in the first place. v1 ships the in-house verifier path; a future ADR may revisit the broker option if the substrate-scaling cost becomes load-bearing and a sufficiently trustworthy broker emerges.

We considered **using transparency-log anchoring** (Sigstore-style append-only public log, blockchain-anchored receipts, etc.) as part of v1's evidence chain. Rejected at v1 scope but committed as a known gap. Transparency-log anchoring strengthens the customer-verifiable property (the customer can detect retroactive manifest tampering by checking the log entry against the published manifest), and the Sigstore ecosystem provides an off-the-shelf option. v1 ships signed manifests without transparency-log anchoring to keep the v1 surface tight; v2 (a future ADR) will commit a specific transparency-log scheme when the operational pattern is validated.

We considered **enforcing attestation on Mirepoix-build venue bundles** (not just Mirepoix-secure). Rejected at the policy layer but supported at the architectural layer. Mirepoix-build is the default venue per ADR-012 and is used for non-confidential workloads; requiring attestation on all Mirepoix-build bundles would impose the per-publication latency budget on workloads that do not benefit from the compliance property. ADR-011 commits enforcement for any bundle that *declares* attestation (which is the operator's choice per the bundle manifest); bundles that omit the attestation field continue to publish without verification. This is the opt-in pattern at the policy layer with the enforcement guarantee at the architectural layer — operators who declare attestation get enforcement; operators who don't, don't pay the latency or operational cost.

## Implementation notes

This ADR commits the architectural decision but does not itself ship the v1 implementation. The implementation ships in a sequence of sub-phases, each tracked in `IMPLEMENTATION-PLAN.md` and gated by the on-loop SDLC pipeline.

The first implementation deliverable is `@mirepoix/attestation` as a new package in the Harness bounded context. Initial scope: the manifest format (JSON Web Signature, schema in `attestation/schema/manifest.v1.json`), the GCP TDX verifier (calls Intel's attestation service via the documented HTTPS endpoints), the audit-trail emission (per ADR-005's reconstructability invariant), and the publication-gate integration point that `@mirepoix/cli`'s publication command consumes. The package exports a `verifyAttestation(bundle, declaredProfile)` function that returns a structured result (`{ verified: true, quote, evidence }` or `{ verified: false, reason }`); the publication gate consumes this and decides on publication.

The second implementation deliverable is `@mirepoix/verify-attestation` as a public npm package (separate from the Harness package because customer-side tooling has different distribution semantics). Initial scope: read a signed manifest from a local file or URL; re-fetch the bundle's content hashes from the local download; re-verify the measurement quote against Intel's attestation service; produce a structured verdict with documented exit codes; ship documentation for customer compliance officers. The package vocabulary aligns with Kirk-real's customer-side verifier per commitment three.

The third implementation deliverable is the `@mirepoix/cli` publication-command integration. `mirepoix bundle publish` runs the attestation gate before any registry push; rejection prints a diagnostic naming the verification failure mode (quote missing, quote invalid, content-hash mismatch, etc.); approval prints the verified manifest's location and the customer-side verification command for the published bundle. The audit log captures the verdict and the verifier's evidence.

The fourth implementation deliverable is the `@mirepoix/acp-server` integration committed in ADR-015. When ADR-015's `@mirepoix/acp-server` lands, the attestation verification path becomes available to ACP clients through a documented surface (probably `request_permission` notification with the attestation evidence, allowing the client's UI to render the compliance state per session). The exact ACP surface is to be locked in the ADR-015 implementation sub-phase, not in ADR-011.

The fifth implementation deliverable is documentation. The Mirepoix-secure runbook (`docs/MIREPOIX-SECURE-RUNBOOK.md`) gains an "Attestation enforcement" section documenting the verification flow, the failure modes, and the operator-side troubleshooting. A new Confluence sibling page in the Platform Engineering space documents the customer-facing compliance posture and cross-links to the GitHub ADR; the cross-link to Confluence PE/111017985 (Kirk-real's customer-verifiable attestation chain) is established as the canonical cross-product reference. The Confluence page is the team-facing surface per the kavara-builder / Confluence / GitHub coordination pattern (memory: feedback_kavara_working_pattern.md).

The empirical validation pattern from PR #5 / PR #18 / PR #2 (kirk) / PR #2 (scotty) / PR #3 (kirk) per ADR-013's multi-agent face-off — the discipline that catches real pre-merge defects — applies to the `@mirepoix/attestation` sub-phase. The package's REVIEW phase dispatches Codex as a parallel reviewer per ADR-013's first commitment; verification-gate logic in particular benefits from adversarial review because the failure modes (quote-verification-bypass, content-hash-mismatch-papering, evidence-truncation) are exactly the kinds of subtle architectural defects a single reviewer might miss.

## Known gaps (v1)

This ADR ships the architectural commitment at the framework layer. Several operational and protocol surfaces are deferred to follow-up sub-phases and follow-up ADRs; v1 is honest about what is **not** committed today.

- **Runtime drift detection is named but deferred (HIGH).** Commitment four explicitly defers runtime drift detection (binary diff against the manifest's content hashes, periodic re-attestation, alert routing). v1 ships publication-time enforcement only. Customers who require runtime drift detection are documented as needing the Mirepoix-secure venue's tighter operational envelope (deny-all-egress + bastion-mediated audit per ADR-010). A future ADR (probably ADR-018 or beyond) will commit a runtime-drift-detection design when the operational pattern is validated.
- **Multi-substrate profiles are reserved for future ADRs (HIGH).** v1 ships GCP TDX only. Azure TDX, AWS Nitro Enclaves, AWS AMD SEV-SNP, and on-premises bare-metal TDX appliances each require their own verifier, their own manifest profile, and their own customer-side verification flow. ADR-010's substrate-extension forward-reference applies; per-substrate ADRs will land as commercial signal warrants.
- **Transparency-log anchoring is deferred (MEDIUM).** v1 ships signed manifests without Sigstore-style or blockchain-anchored transparency-log entries. The customer-verifiable property is strong against transit tampering and at-rest tampering of the bundle, but weak against retroactive manifest reissuance by a compromised Mirepoix-controlled key. A future ADR will commit a specific transparency-log scheme; Sigstore is the leading candidate but not yet locked.
- **Customer-side verification tool ships as architecture-committed, implementation-deferred (MEDIUM).** Commitment three commits the `@mirepoix/verify-attestation` shape and the vocabulary alignment with Kirk-real's customer-side verifier; the implementation lands in a sub-phase, not in this ADR. The dependency for ADR-015's v0.2.0 launch is that the tool is shippable in time for the Mirepoix Cloud public release; if the tool is not ready, v0.2.0 ships without it and customers receive the manifest but cannot run the cross-check until the tool catches up. The risk is named here so it is not surprising at v0.2.0 release time.
- **Pre-ADR-011 bundle migration is policy-only (MEDIUM).** Commitment five accepts that observational bundles continue to work as-is; there is no migration script. Customers operating on pre-ADR-011 bundles who want enforced attestation must republish through the new pipeline. For customers with significant pre-ADR-011 bundle inventory this is operational friction; v1 accepts the friction as the price of clean architectural commitment, and a future operational sub-phase may build a bulk-republish tool if customer demand warrants it.
- **Intel TDX attestation service availability is an external dependency (MEDIUM).** Publication latency is bounded by Intel's service response time; service outages block all attestation-declared publications until the service recovers. v1 does not commit a circuit-breaker or fallback path; a future operational sub-phase will commit a documented degradation mode (e.g., queue publications for verification when the service is unavailable, with operator notification). The risk is named here so it is not surprising at v0.2.0 production deployment time.
- **Per-substrate verifier maintenance cost scales with substrate count (LOW).** As multi-substrate ADRs land, each new substrate adds a verifier the team must keep current against the substrate's attestation-service evolution. v1 has bounded burden (one substrate). The cost projection for n substrates is named here so it can inform the per-substrate-ADR decision criteria when the time comes.
- **The `@mirepoix/attestation` package's API surface is named, not specified (LOW).** Implementation note one names the `verifyAttestation(bundle, declaredProfile)` function signature but does not specify the full type system. The TypeScript types will land with the package's first sub-phase per the ADR-001 minimal-core discipline; reviewers should not block this ADR on API specification.
