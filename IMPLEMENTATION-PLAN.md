# Mirepoix — Implementation Plan

Owner: John Edge (CTO, Kavara)
Status: Draft v0.1
Date: 2026-05-06

## What we're building and why

Mirepoix is Kavara's coding agent harness. It is the loop that takes a model, a set of tools, and a body of context, and turns them into useful work on a codebase. We are building our own rather than adopting Claude Code, OpenCode, Amp, or Factory because the harness is the operating system of an agent-driven engineering org and we are not willing to outsource that operating system to a vendor whose roadmap, telemetry posture, context-handling decisions, and security model we do not control. Mario Zechner's talk *Building Pi in a World of Slop* — Mirepoix's spiritual ancestor under its original Pi placeholder name — is the architectural spine of this implementation. We adopt his core thesis — that current coding agents are not in their final form, that the harness must be malleable by the operator and by the agent itself, and that the bottleneck on quality is human discipline rather than model capability.

Concretely, Mirepoix at Kavara is a TypeScript-first harness with a tiny core, four base tools, a typed extension API, an in-process event bus, hot-reloadable extensions, and a context surface that is fully visible to the operator at all times. Every byte of context that goes to the model is visible, attributable, and modifiable. There are no hidden system reminders, no silent tool-definition mutation, no automatic tool-output pruning, and no behind-the-back LSP injections. If the harness is going to lobotomize the model, it does so transparently and only because we asked it to.

## Mirepoix as a Software 3.0 operating-system distribution

The framing for this work is more ambitious than "Kavara's coding agent harness." If we take Karpathy's Software 3.0 thesis at face value — that the LLM is the new operating system, with the context window as RAM, reasoning as CPU, RAG as filesystem, MCP-like tool protocols as the driver layer, and skill manifests as `package.json` — then what Mirepoix actually is, in product terms, is a *distribution* of that operating system. Mirepoix-base is the kernel and drivers; Kavara-Mirepoix is the Kavara distribution of the kernel; Customer-X-Mirepoix instances are remixes of that distribution for specific deployments. This is the same shape Linux + Ubuntu + Ubuntu-for-IoT takes, and the same shape BSD + macOS + macOS-Server takes. We adopt it deliberately.

The Mirepoix metaphor is the in-house version of this framing. Mirepoix-base is the Mirepoix — the small invariant base of foundational ingredients used in every dish. Kavara-Mirepoix is the Kavara sofrito — the regional variant that gives every Kavara-prepared dish its identifiable character. Customer-X-Mirepoix is the actual finished dish — branded, regional, ready to serve. Each layer composes onto the one below; each carries its own license posture; each is versioned independently. ADR-007 specifies the architectural mechanism that enforces this layering, the per-extension license-tagging contract, and the build-time bundler that prevents mixing the wrong content across layers.

This framing also tells us where Mirepoix's value to the Model-as-a-Service business actually lives. Karpathy's verifiability principle — that AI excels where verification is cheap and struggles where it is not — concentrates Mirepoix's leverage on engineering work for Tiberius, Uhura, Kirk, and the Red Hat collaboration, where compilers and tests give us cheap verification, and on customer-deployment operations where `oc`, `kafka-get-offsets`, and runbook outcomes give us mid-cost verification. Mirepoix gives less leverage on low-verifiability tasks like creative comms, and we should be honest that the system is not trying to be useful everywhere. The scarcity-shift thesis from the same source — that taste, evaluation design, and architectural judgment are the rare resource — tells us what Kavara is actually selling: not Kirk-the-model in isolation, but Kirk-on-substrate-with-engineering-rigor. Every Kavara-Mirepoix extension is a piece of that rigor preserved as code; every Customer-X-Mirepoix instance is that rigor applied to a specific deployment.

## Non-goals

We are not building a marketplace. We are not building a hosted product. We are not building a UI for end-users outside Kavara. We are not building anything that requires us to retain user content. We are not implementing permission dialogs, sandboxing layers, or policy engines inside the core — those are operator concerns that live in extensions or in the surrounding infrastructure. We are not optimizing for parity with Claude Code's feature surface; if we end up missing a feature, the answer is to write an extension or to live without it.

## Architectural principles

The following principles are load-bearing. Every implementation decision is checked against them before it lands, and each is captured more rigorously in the ADRs that accompany this plan.

The harness is minimal. The core is four packages and not one more. The system prompt is a handful of lines because post-trained models already know what a coding agent is. The base tool set is bash, read, write, edit, and that is it. Anything else is an extension.

Context is owned by the operator. The harness emits the prompt, tool list, message history, and tool results in a form that the operator can inspect, log, redact, and mutate. The harness never silently injects content into the context window. It never silently changes a tool definition between sessions. It never prunes tool output without the operator opting in.

Extensions are TypeScript modules and nothing more. They register tools, prompts, shortcuts, event listeners, providers, and compaction strategies through a typed API. They are loaded by file path or by NPM package name. They hot-reload during a live session. They live anywhere on disk or in any package registry; the harness does not care.

The event system is a real bus. Hooks do not spawn child processes. Listeners are functions registered against typed events, and the cost of an event with no listeners is one comparison and one branch. This is non-negotiable because we will fire events at a rate that a per-event subprocess model cannot tolerate.

Bash is unrestricted by default. Security is the operator's responsibility, not the harness's. We expect operators to run Mirepoix inside containers, sandboxes, or trusted environments. We do not pretend that prompting the user with "Allow rm -rf?" is a security mechanism, and we will not build that into the core.

The agent can modify itself. Mirepoix ships with documentation and example extensions inside the repository so that the agent can read them and write new extensions during a session. This is the single feature that compounds. Every other harness ages; Mirepoix gets younger with every session because it can grow the surface it needs.

Distribution is NPM. Skills are markdown files. We adopt the emerging skills standard rather than inventing a Kavara-specific extension format on top of it. We do not build a marketplace, an extension store, or a curation layer. NPM and GitHub are sufficient.

The distribution is layered. Mirepoix-base is open-source on public NPM. Kavara-Mirepoix is the curated Kavara bundle; its public surface lives at `github.com/UlyssesModel/kavara-mirepoix` and contains only extensions that have been deliberately tagged for public release. Internal extensions live in a parallel private repository under the same organization. Customer-X-Mirepoix instances are private per-customer remixes. Every extension carries a distribution tag — `internal` (the default), `customer-licensed`, or `public` — and the bundler enforces the contract at build time. Conservative-by-default with case-by-case opt-in to public is the operating posture. ADR-007 captures the full mechanism.

## Package layout

Mirepoix consists of four published packages under a single monorepo. The first is `@mirepoix/ai`, which is a thin abstraction over model providers — one interface that takes a list of messages and a list of tool definitions and returns either a text response or a tool call, with streaming and cancellation. It owns nothing about agents, tools, or context strategy. Second is `@mirepoix/core`, the agent loop. It is a typed event bus, a session model, a tool dispatcher, and a `while`-loop that calls the AI package, dispatches tool calls, and emits lifecycle events. It does not know it is a coding agent. Third is `@mirepoix/coding`, the coding-agent skin. It contributes the four base tools, the system prompt, the skills loader, and the default compaction strategy. It depends on `@mirepoix/core` and `@mirepoix/ai`. Fourth is `@mirepoix/cli`, the entry point — argument parsing, terminal I/O, session persistence, and extension discovery. It depends on `@mirepoix/coding`.

This decomposition is deliberate. A team that wants Mirepoix-as-a-library for a non-coding application uses `@mirepoix/core` and writes its own skin. A team that wants to embed the coding agent in a different shell uses `@mirepoix/coding` and writes its own CLI. The CLI is replaceable; the core is not.

## Phased build

The plan is sequenced so that we have something working at every milestone and never let the codebase grow beyond what one engineer can hold in their head. This is a direct response to Act Three of the talk: the cost of letting an agent generate a system you can no longer review yourself is unbounded, and the only honest mitigation is to keep the system small enough that you can.

Phase Zero is a spike. One week. A single TypeScript file that calls Anthropic's API, exposes a `bash` tool, and runs a tool-calling loop until the model stops calling tools. No packages, no event bus, no extensions, no skills. The deliverable is a script that successfully runs `git status` and reports the result in conversation. The point is to validate the model's behavior with a near-empty system prompt and to feel where the friction lives before we commit to a structure.

Phase One is the minimal core. Two to three weeks. We split the spike into the four packages, introduce the typed event bus, the session model, and the four base tools. We add the skills loader (markdown files in a `skills/` directory), session persistence to disk, and a basic terminal UI. No extensions yet, no hot reload, no self-modification. The deliverable is a CLI that an engineer at Kavara can install via `npm i -g @mirepoix/cli`, point at a repo, and use to perform real coding tasks. It is feature-poor and that is the point.

Phase Two is the extension API. Two to three weeks. We define and ship the typed extension interface, the loader (path-based and package-based), and hot reload. We document the API in a way that is digestible by the agent itself — that is, with worked examples that the model can read and adapt. We migrate one or two pieces of the coding-agent skin into extensions to validate that the API is sufficient. The deliverable is the ability for any engineer at Kavara to drop a TypeScript file into a config directory and have it modify the harness on the next session start, plus the ability to hot-reload an extension without leaving the session.

Phase Three is self-modification. One to two weeks, but mostly testing and tightening rather than new code. We ship the extension documentation in the repository in a form the agent can read, expose a "develop an extension" subroutine, and tune the prompts and skills so that the model reliably writes useful extensions on request. The deliverable is a video or transcript of a Kavara engineer asking Mirepoix to add a feature, watching Mirepoix write the extension, and using the new feature in the same session.

Phase Four is the first composed Kavara-Mirepoix bundle. Two to three weeks. With Mirepoix-base now operable and self-modifying from Phases One through Three, Phase Four assembles the first version of Kavara-Mirepoix — the curated middle layer of the Mirepoix. The deliverables are the public bundle repository (`github.com/UlyssesModel/kavara-mirepoix`) populated with the first set of `public`-tagged extensions (the cascade-routing model selector, the architecture-review meta-agent with domain experts, the multi-persona reviewer, the OpenShift connector, the AGENTS.md template tuned for Kavara conventions, the slash-tree session navigation extension, the footer with model + context-fill + branch + tool counter, the autonomy-preset definitions for `careful`, `trusted`, and `autonomous`); the parallel private repository for `internal` extensions (the KServe-on-OCP provider for Kavara's own infrastructure, the Tiberius/Uhura/Kirk connectors, the substrate-matrix-aware skills, the customer-deployment runbooks); and the `mirepoix bundle` command that composes both into a working harness for Kavara engineers. Kavara-Mirepoix v0.1 is the deliverable that turns the open base into the harness Kavara engineers actually run.

Phase Five is per-customer remix. Open-ended, project-by-project. As Kavara MaaS engagements move from POC to production — federal customers, Databank DL360 deployments, NY5, Red Hat collaboration sites — each one gets its own Customer-X-Mirepoix instance: a private repository referencing a specific Kavara-Mirepoix version, layered with customer-specific extensions, skills, configs, and an autonomy preset matched to the customer's posture. Customer-deliverable bundles are signed, ride the same distribution channel as the Kirk appliance itself, and become part of the MaaS deliverable. We do not pre-build customer remixes; we build them in response to real engagements, and we treat the time-to-first-customer-Mirepoix-remix as a leading indicator of MaaS go-to-market readiness.

Phase Six is hardening, in the original sense of the earlier draft. Open-ended. Custom compaction strategies for long sessions, additional providers (OpenAI / Gemini / self-hosted KServe / on-prem local), session telemetry exports, structured logging, observability dashboards, prompt-cache integration. We do not pre-build for this phase; we let real usage at Kavara and at customer sites select what gets prioritized.

## What we are explicitly deferring

Sub-agents are an extension, not a core feature. Plan mode is an extension. MCP support is an extension. A web UI is not on the roadmap. Multi-tenant support is not on the roadmap. Granular permission systems are not on the roadmap. If a Kavara team wants any of these, the answer is to write the extension or to ask Mirepoix to write it.

## Success metrics

We will judge the project on four things. First, system mass — the number of lines of code in the core packages. We target under five thousand lines for `@mirepoix/core`, `@mirepoix/ai`, and `@mirepoix/coding` combined at the end of Phase One, and we treat unbounded growth as a regression. Second, context transparency — every byte sent to the model is logged, attributable, and inspectable, and we measure this by being able to reconstruct any past session's full prompt sequence from the session log. Third, time-to-extension — the elapsed time between an engineer wanting a feature and being able to use it. We target under one hour for any extension that does not require a new tool primitive, achieved by having the agent write the extension. Fourth, terminal-bench score — we target a top-ten placement on the Terminal-Bench leaderboard within six months of Phase One landing, mostly as a sanity check that we have not built something subtly broken.

## How this plan relates to Act Three of the talk

Mario's third act is a critique of how agents are used, not how they are built. We take it seriously inside the plan because it shapes our review and merge discipline more than any architectural choice. We will not let agents merge to `main` unsupervised. We will not let agents introduce abstractions, defense-in-depth wrappers, or backward-compatibility shims into the core packages without an ADR justifying the addition. We will read every line of code that lands in `@mirepoix/core` and `@mirepoix/ai` regardless of whether a human or an agent wrote it. We will let agents write extensions, fixtures, reproductions, and docs with much lighter review, because that is exactly the kind of "non-mission-critical" work the talk identifies as the right place to apply them.

Mirepoix-on-Mirepoix sessions run under the `mise-en-place` agent contract, a private behavioral spec at `github.com/UlyssesModel/mise-en-place` that adapts Karpathy's four agent-behavior principles and adds an eight-mode operator-elicited posture system (`build`, `explore`, `harden`, `ship`, `firefight`, `review`, `migrate`, `refactor`), and is the specific tool we use to enforce the discipline this section describes. The contract is starter-state and expected to leak before it hardens; it is not yet a Mirepoix-base or Kavara-Mirepoix component, and inclusion at higher levels — vendoring into the bundle, ADR-010 adoption, runtime integration with the footer-as-surface and listener-vs-gate hook patterns — is gated on accumulated evidence from real sessions rather than a planning decision now.

## Architecture decision records

The decisions that lock in the shape of Mirepoix are captured as ADRs in the `adrs/` directory. Each ADR follows the Michael Nygard format — context, decision, consequences, alternatives considered. The current set is ADR-001 through ADR-008. ADRs are immutable once accepted; if a decision changes, a new ADR supersedes the old one and the old one is marked as such.

ADR-001 covers the four-package decomposition. ADR-002 covers the tool surface and the bash-unrestricted security posture. ADR-003 covers the extension model and self-modification. ADR-004 covers the event bus and the rejection of hook-spawn-process designs. ADR-005 covers context ownership and the observability commitments. ADR-006 covers distribution, skills, and the no-marketplace stance. ADR-007 covers the three-layer distribution model (Mirepoix-base, Kavara-Mirepoix, Customer-X-Mirepoix) and the per-extension license-tagging contract enforced by the bundler. ADR-008 covers model routing, provider abstraction, and substrate-aware self-hosted serving — cascade and task-class routing policies, four provider tiers including the multi-substrate Qwen3-Coder fleet on GCP / Azure / AWS-AMD-SEV-SNP plus on-prem appliances, and the eval-driven `mirepoix sweep` harness that gates routing-default decisions.

Queued for the next round, in priority order: a v2 self-review pass on ADRs 001-006 against the rigor bar set by the `ts_sor_base` v3 architecture review (separating structural claims that are defendable now from quantified claims that need measurement). A smaller ADR on session tree navigation (the `slash tree` pattern from Lucas Meyer's talk, plus the listener-vs-gate-hook distinction needed for til-done and damage-control patterns). And the first concrete eval-suite landing in `kavara-mirepoix-internal/eval-suites/coding/` so that ADR-008's sweep methodology is operational rather than aspirational.

## Open questions

A handful of decisions are still open and need to be closed before Phase One begins. We have not yet picked a TypeScript runtime — Bun is fastest and most ergonomic, Node is most familiar to candidates, Deno is the most opinionated about security. We have not yet committed to a session storage format — JSON files per session is simple and inspectable, but a single SQLite database is cheaper to query and resists partial-write corruption. We have not yet decided whether the AI package wraps the Anthropic SDK or hits the API directly; the SDK adds a dependency and a moving target, but the API surface is small enough that we may not save much by going direct. These will be closed in dedicated ADRs before any Phase One code lands.
